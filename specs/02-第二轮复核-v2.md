## 2. 🔴 P0 新问题一:屏 10 的图标网格违背 iOS 现实

### 问题

屏 10「添加例外」的服务网格里有:**微信、Telegram、Instagram、YouTube、支付宝、Chrome、淘宝**。

选中后下方写「**Telegram 的流量:** 总是直连 / 总是走线路 / 拦截」。

**两个错误:**

**① Chrome 根本不能作为分流对象。**
iOS 上不存在「Chrome 的流量」这个概念。Chrome 访问的是成千上万个不同网站,分流是**按访问的网站**走的,不是按发起的 App。给 Chrome 设「总是走线路」在技术上无法实现(见 v1 文档 1.2 C 类:iOS 不支持按 App 分流)。

**② 图标比文字更有说服力。**
P0-3 我们把文案从「应用」改成了「网站与服务」,但这个 App 图标网格在视觉上又教会用户「我可以按 App 设置」。**文案改了、视觉没改,等于没改。**

**③ 微信 / 支付宝 / 淘宝 出现在这里是误导。**
智能分流本来就把它们直连了。把它们放在「添加例外」的第一屏,等于暗示用户"你需要给微信做点什么"。用户点进去只会困惑。

### 要求修改

1. **删除 Chrome**。浏览器不是分流对象
2. **删除微信 / 支付宝 / 淘宝**(以及一切智能分流已正确处理的国内服务)
3. 网格里只保留**有独立域名、且确实需要特殊处理**的服务
4. 图标旁的措辞从 App 图标观感改为服务观感,文案统一为「**XX 的网站与服务走哪里**」而不是「XX 的流量」
5. 按分类组织,不要平铺(见 §5 提示词 8-v2)

## 3. 🔴 P0 新问题二:屏 9 与屏 9-v2 必须合并,且 v2 有两处倒退

### 现状:两套并存

| | **屏 9**(标 ★「全 App 核心页」) | **屏 9-v2** |
|---|---|---|
| 智能分流可视化 | ✅ **「流量去向」三行图解** | ❌ 删了 |
| 例外清单 | ✅ **「我的例外」**,默认 2 条 | ❌ 退化为「服务分流」,默认 5 条 +「全部 26 项」 |
| 场景规则入口 | ❌ 无 | ✅ 有 |
| 高级入口 | ✅「高级设置(自定义规则)· 普通用户无需进入」 | ✅「规则集 · 自定义规则」 |

**开发看到两个屏 9 会不知道做哪个。必须合并成唯一的 v3。**

### 倒退一:v2 删掉了「流量去向」图解 —— 这是整套设计最好的东西

屏 9 的这三行:

```
国内网站与服务   微信、淘宝、银行          →  🏠 直连   不绕路,速度快
国外网站与服务   YouTube、Instagram、ChatGPT →  🚀 走线路
广告与跟踪       弹窗、行为追踪器           →  🚫 拦截   今日已拦截 1,284 次
```

**它用三行就让零基础用户看懂了「智能分流」到底在干什么**,而且举了具体例子。这解决了「用户不知道这个 App 在替我做什么决定」的根本焦虑,比任何文案都有效。

**v3 必须保留它。**

### 倒退二:v2 把「我的例外」改回了「配置清单」

屏 9 原本的心智是对的:

- 标题「**我的例外**」
- 副文案「**想让某个服务或网站特殊处理?**」
- 默认只有 2 条

这正是正确的方向 —— **例外清单**,不是配置清单。

而 v2 改成:

- 标题「服务分流」+ 副文案「给常用服务指定专属线路」
- 默认铺 5 条 +「**全部 26 项 ›**」

**「全部 26 项」是在告诉用户"你有 26 个东西要管"**,压力感回来了。而智能分流本来已经把这 26 个都处理好了。

> **判断标准:预设的门槛不是"常用",而是"不预设就会出问题"。**

### v3 应该长什么样

合并两版的优点:

```
分流
决定哪些流量走线路、哪些直连

[智能分流 · 推荐]  ← 选中
[全局代理]
[全局直连]

流量去向                          ← 保留屏 9 的图解,不要删
  国内网站与服务  →  🏠 直连
  国外网站与服务  →  🚀 走线路
  广告与跟踪      →  🚫 拦截  今日已拦截 1,284 次

服务例外    想让某个服务特殊处理?   ← 用屏 9 的心智 + v2 的胶囊表达
  Claude    🇺🇸 美国·洛杉矶-01 🔒
  ChatGPT   🇺🇸 美国·洛杉矶-01 🔒
  Gemini    🇺🇸 美国·洛杉矶-01 🔒
  X         🇺🇸 美国·洛杉矶-01 🔒
  TikTok    🇯🇵 日本·东京-02 🔒
  + 添加例外                        ← 点进去是「推荐库」

场景规则                 已启用 2 条 ›   ← 保留 v2 新增

高级设置(自定义规则)›  普通用户无需进入
```

**去掉「全部 26 项」这个入口。** 例外就是例外,有几条显示几条。

## 4. 🟡 其余问题

### P1-A:TikTok 应该是「锁定」不是「自动」

屏 13 现在是 `🇯🇵日本 · 自动 🔄`。

但 TikTok 属于**必须锁定**的 5 条之一:地区直接决定内容池,而且 **IP 乱跳会让推荐内容混乱、账号触发异常**。「自动」= 会换 IP = 正是要避免的。

→ 改为 `🇯🇵日本 · 东京-02 🔒`

### P1-B:X (Twitter) 完全缺失

屏 13「社交」组只有 Telegram 和 Instagram。**X 是风控最激进的服务之一**,IP 跳动频繁触发手机 / 邮箱验证,严重时锁号 —— 和 AI 服务同一个逻辑,**必须预设锁定**。

→ 加入 X,`🇺🇸美国·洛杉矶-01 🔒`,建议放在 AI 组下方或社交组首位

### P1-C:屏 7「每 24 小时自动更新一次」承诺了 iOS 给不了的保证

**iOS 没有可靠的后台定时任务。** App 被挂起/杀掉时不会自己去拉订阅;系统的后台刷新不保证准时,用户还能在设置里直接关掉。实际更新几乎都发生在**用户打开 App 时**。

→ 改为「**约每 24 小时更新一次,或在你打开 App 时**」

### P1-D:屏 7 缺自建订阅的降级态

自建线路(CF Workers 等)**拿不到流量配额和到期日** —— 那是机场的商业模型,不是技术属性。现在的订阅卡只有「有流量有到期」这一种完整态。

→ 补一个精简态:只有订阅名 + N 条线路 + 上次更新时间,**进度条与到期整块隐藏**。不要显示「未知」「—」或空进度条。

> 自建用户(尤其 CF 那批)在早期种子用户里占比不低,他们打开看到一排空进度条,第一印象就是产品很糙。

### P1-E:屏 7 缺「数据陈旧」态

拉订阅失败是常态(网络差、机场挂了、用户几天没开 App)。这时还理直气壮显示「剩 61.5 GB」就是在骗人。

→ 数字灰化 + 「3 天前的数据 · 点击重试」

### P1-F:屏 6 分组头有「剩 47 天」但没有剩余流量

用户最常检查的是**流量会不会超**,现在得点进「管理」才看得到。

→ 分组头右侧改为 `剩 61.5 GB · 47 天` + 一条 2px 细进度条(**兼做分组分隔线,不额外占高度**)
→ 剩余 <10% 或 <7 天时转红 + 警告图标
→ 自建订阅:整块隐藏

### P1-G:屏 6 的「2x」倍率被做得太轻

`🇺🇸 美国 洛杉矶-01 [2x] [Hy2] 178ms` —— 倍率和协议做成了同级的小灰标签。

但**倍率直接决定扣多少流量,比协议重要得多**。协议(Trojan/VLESS/Hy2)对普通用户毫无意义,倍率才是钱。

→ 节点行改双行(见提示词 10),倍率放第二行**用警告色**:`Hy2 · 2x 倍率`
→ 只显示解析出 ≠1x 的(内核没有倍率字段,靠解析线路名,"没写"和"1x"分不清)

### P2-A:屏 4 的流量预警要和数据新鲜度配合

「⚠️ 流量剩余 8%,点击查看」这个预警很好。但如果这个数字是 3 天前拉的,预警可能不准。

→ 数据陈旧时,预警条也要弱化或标注时间

### P2-B:屏 14 地区网格的弱高亮(v1 提示词 9,未做)

选中「🔒锁定具体线路 · 美国 洛杉矶-01」时,上方地区网格里的「美国」没有任何呼应,用户会疑惑「我到底选没选美国」。

→ 给「美国」加**弱高亮**(浅色描边),表示"你锁定的线路属于该地区",但必须与「选中该地区自动」的选中态明显不同

```
1. 正常:如上
2. 告警:剩余 <10% 或 <7 天 → 数字与进度条转红 + 警告图标
   (这是用户唯一真正在意的时刻)
3. 自建订阅(拿不到流量与到期):只显示「我的自建 3 条」,
   进度条与到期整块隐藏。不要显示「未知」「—」或空进度条
4. 数据陈旧:数字灰化 + 「3 天前的数据 · 点击重试」
```

### 提示词 F — 屏 7 订阅管理的文案与降级态(P1)

```
修改 Swell 屏 7「订阅管理」。

问题一:文案承诺了 iOS 给不了的保证
现在写「每 24 小时自动更新一次」。但 iOS 没有可靠的后台定时任务:
App 被挂起时不会自己去拉,系统后台刷新不保证准时且用户可关闭。
实际更新几乎都发生在用户打开 App 时。
→ 改为「约每 24 小时更新一次,或在你打开 App 时」

问题二:缺自建订阅的降级态
自建线路(如 CF Workers)拿不到流量配额和到期日——那是机场的商业模型,
不是技术属性。现在只有「有流量有到期」这一种完整态。
→ 补精简态卡片:只有订阅名 + N 条线路 + 上次更新时间,
  流量进度条与到期整块隐藏。不要显示「未知」「—」或空进度条。

问题三:时间戳要和数字绑定
流量数字是机场服务端的账,机场自己有几分钟到几十分钟延迟,
且本地正在跑的流量不会实时反映进去。用户会疑惑
「我刚下了 10GB 怎么数字没动」。
→ 「2 小时前更新」不能只放在角落,要和流量数字视觉上关联

问题四:补「数据陈旧」态
拉取失败超过 24 小时时,数字灰化 + 「3 天前的数据 · 点击重试」。
拉不到订阅是常态(网络差、机场挂了、用户几天没开 App),
这时还理直气壮显示「剩 61.5 GB」就是在骗人。
```

## 7. 附录:内核事实速查(设计红线)

| 想要的 | 现实 |
|---|---|
| **iOS 按 App 开关代理** | ⛔ **做不到**。iOS 不告诉隧道流量属于哪个 App。只能按域名/IP 分流 |
| 自动挑"能解锁 Netflix"的线路 | ⛔ 内核只测延迟,不测解锁。要做得自建探测系统 |
| 自动挑"带宽最大"的线路 | ⛔ 只有延迟,没有带宽概念 |
| 又要自动选最快、又要 IP 不变 | ⛔ **互斥,二选一** |
| 屏蔽视频里的贴片广告 | ⛔ 只能按域名拦。YouTube 广告与正片同域名 |
| 内核知道线路在哪个国家 | ⛔ **不知道**。地区是 App 解析线路名猜的,会猜错 |
| 内核有"倍率"字段 | ⛔ **没有**。只能解析线路名,"没写"和"1x"分不清 |
| 流量 / 到期 | ✅ 来自订阅的 HTTP 响应头,**只在拉订阅那一刻拿得到** |
| 按 Wi-Fi 名称 / 蜂窝 / 低数据模式分流 | ✅ **iOS 支持**,已在屏 17 用上 |

**整体评价:这一轮的 🔒/🔄 图标语言、屏 15 的模式覆盖处理、屏 16 的非阻断冲突提示,都做得很好。剩下的问题集中在两处:视觉上还在暗示「按 App 分流」,以及 v2 丢掉了 v1 最好的两个设计。**
