# Swell 分流功能 — 内核能力说明 & UI 修改需求

**给:Design**
**版本:v1 · 基于 sing-box 1.13.14 官方文档核实**

---

## 0. 你需要知道的背景

Swell 的代理内核用的是 **sing-box**。你在 UI 上设计的每一个"分流"开关,最终都会被翻译成内核的一张**规则表**。

这意味着:**UI 只能表达内核能表达的东西。** 如果 UI 承诺了内核做不到的事,不是工程偷懒,是物理上做不到——用户会因为预期落空来投诉。

这份文档分三部分:

1. 内核到底能做什么(你的设计边界)
2. 现有设计稿哪里越界了 + 怎么改
3. 可以直接复制给设计工具的提示词

---

## 1. 内核的工作方式(心智模型)

### 1.1 一句话原理

> 一张**从上到下**的规则表。流量来了,逐条比对,**第一条命中的规则决定它去哪**。一条都不中,走"兜底"。

每条规则 = **【匹配什么】** + **【去哪】**

这个"顺序"很重要,后面会讲为什么它影响你的设计。

### 1.2 能匹配什么(能按什么维度分流)

#### A 类:按**目的地** — 主力,iOS 完全可用

| 维度 | 说明 | 例子 |
|---|---|---|
| 域名 | 精确 / 后缀 / 关键词 / 正则 | `claude.ai` |
| **服务分类集** | 打包好的域名集合,最常用 | "所有 YouTube 域名" |
| IP / IP 段 | | `192.168.0.0/16` |
| **国家(geoip)** | 按 IP 归属国 | "所有中国 IP" |
| 端口 | | `443` |
| 协议 | 靠嗅探识别 | TLS / QUIC / BitTorrent |
| 是否私有网络 | 局域网地址 | 用于"私有网络直连" |

**你现在稿子里的服务分流(Claude / YouTube / Netflix…)全部属于这一类**,靠的是匹配这些服务的域名集合。

#### B 类:按**环境** — iOS 可用,**但你现在完全没用上,是机会点**

| 维度 | 说明 | iOS 支持 |
|---|---|---|
| 网络类型 | Wi-Fi / 蜂窝 / 以太网 / 其他 | ✅ |
| 是否计费网络 | 蜂窝数据、个人热点 | ✅ |
| 是否低数据模式 | 系统的"低数据模式"开关 | ✅(苹果专有) |
| **Wi-Fi 名称(SSID)** | 当前连的是哪个 Wi-Fi | ✅ |
| Wi-Fi BSSID | 具体路由器 | ✅ |
| 当前模式 | 智能 / 全局 / 直连 | ✅ |

#### C 类:按**来源 App** — ⛔ iOS **做不到**

| 维度 | 支持平台 |
|---|---|
| 进程名 / 进程路径 | 仅 Linux / Windows / **macOS** |
| Android 包名 | 仅 Android |
| **iOS** | **完全不支持** |

> **这是最重要的一条限制。** iOS 系统不会告诉 VPN 隧道"这条流量是哪个 App 发出的",而且面向普通 App 的"分应用 VPN"只在企业 MDM 环境才有。
>
> 所以 **iOS 上永远无法做出"某某 App 走代理 / 某某 App 直连"的开关**。
>
> 我们能做的是**按目的地**分流。因为 App 通常访问自己特征鲜明的域名(TikTok 连 TikTok 的域名),效果上"像"按 App,但本质不是。两个 App 访问同一域名时无法区分。

### 1.3 能去哪(去向)

| 去向 | 说明 | IP 稳定吗 |
|---|---|---|
| 直连 | 不走代理 | — |
| **某条具体线路** | 锁定单个节点 | ✅ **不变** |
| **一组线路自动选最快** | 定期测速,谁快用谁 | ❌ **会变** |
| 一组线路手动选 | 用户自己挑,挑完就固定 | ✅ 不变 |
| 拦截 | 直接掐断(可选"回绝"或"静默丢弃") | — |
| 链式 | 先经过 A 再经过 B | — |

### 1.4 ⚠️ 重要:内核里**没有"地区"这个概念**

这条单独拎出来讲,因为它直接影响「指定地区」的设计。

内核只认三样东西:**单个节点**、**一组节点自动选最快(urltest)**、**一组节点手动选**。

它**完全不知道"洛杉矶-01"是美国的**。

所以「指定地区 = 美国」是 **App 自己造出来的概念**,实现方式是:

```
1. App 解析线路名称(🇺🇸 / US / 美国 / 洛杉矶 …)→ 猜它属于哪个地区
2. App 把猜出来是美国的线路,打包成一个"自动选最快"的组
3. 路由规则指向这个组
```

**这带来三个设计后果,必须在 UI 上体现:**

| 后果 | 对设计的要求 |
|---|---|
| **归类是猜的,一定会猜错/猜不到**(机场命名毫无标准) | 必须有「未识别地区」兜底 + 允许手动指定。**屏 14 里那句"3 条线路未能识别地区"就是干这个的,保留** |
| **每个地区组都会在后台持续测速**(耗电、占内存,iOS 限制严格) | 只为**用户真正选了的**地区建组。屏 14 展示 6 个地区按钮 ≠ 要建 6 个组 |
| 多个服务选同一地区时,**共用同一个组** | 纯工程实现,UI 无感知,但要写给工程 |

> **给工程的备注**(设计师可跳过):地区组用 `urltest`。默认参数 `tolerance` 只有 50ms,太灵敏会来回跳,地区组建议调到 200–300ms;`interval` 默认 3m 可适当拉长;`interrupt_exist_connections` 默认已是 `false`,保持不变(否则切线路会掐断正在流式输出的 AI 对话)。

### 1.5 订阅数据(流量/到期/倍率)从哪来

**内核完全不管订阅。** 这些全是 App 层的活,而且**来自两个不同的地方**:

#### A. 流量 / 剩余 / 到期 → 来自 **HTTP 响应头**

拉订阅链接时,机场服务器会回一个约定俗成的头:

```
subscription-userinfo: upload=455727941; download=6603013641; total=107374182400; expire=1788220800
```

- 已用 = `upload` + `download`
- **剩余 = `total` − 已用**
- 到期 = `expire`(Unix 时间戳)。**不返回或为 0 = 不过期**

顺带这几个头也有用:

| 头 | 用途 |
|---|---|
| `profile-update-interval` | 机场建议的更新间隔(小时),可作"自动更新"默认值 |
| `profile-web-page-url` | 用户面板地址 → 可做成「打开面板」按钮 |
| `content-disposition` 里的 filename | **订阅名可自动填**,不用让用户手输 |

> ⚠️ 这是**约定俗成,不是标准**。主流面板(V2board / Xboard / SSPanel)都支持,但不保证。必须能优雅处理"头缺失""total=0""expire=0"。

#### B. 倍率 → **没有标准字段,只能解析线路名**

不管 Clash YAML、VLESS 分享链接还是 sing-box JSON,**线路定义里都没有"倍率"字段**。机场只能写进名字:

```
美国 洛杉矶-01 [2x]  ·  香港 IEPL-01 (1.5倍)  ·  US-LA-01 x2  ·  洛杉矶 2.0×
```

**设计结论:只显示解析出 ≠1x 的。** 因为"名字里没写"和"就是 1x"根本分不清,绝大多数又都是 1x。默认不标,解析到才标黄 —— 与「只标记异常」同一原则。

#### C. 自建线路(CF Workers 等)→ 拿不到,**整块隐藏**

两种情况都没有:导入的是单条分享链接(压根没 HTTP 请求)、或自建订阅不返回该头。而且自建**本来就没有"倍率"和"配额"概念**——那是机场的商业模型,不是技术属性。

> **降级要求:** 流量进度条、到期天数**整块隐藏**。不要显示"未知"/"—"/空进度条。

#### D. 什么时候更新?—— **iOS 有硬约束,影响文案**

**`subscription-userinfo` 只在 GET 订阅 URL 的那一刻拿得到。** 没有独立的"查流量"接口。所以:

> **数据新鲜度 = 上次拉订阅的时间。更新订阅 = 更新这些数字,是同一件事。**

而 iOS **没有可靠的后台定时任务**。App 挂起/被杀时不会自己去拉。`BGAppRefreshTask` 由系统调度,**不保证准时**,用户还能在系统设置里关掉。

实际更新只发生在:

| 时机 | 可靠性 |
|---|---|
| App 启动 / 从后台回前台 | ✅ 可靠,**这是主力** |
| 用户手动点刷新 | ✅ 可靠 |
| BGAppRefreshTask | ⚠️ 尽力而为,不保证 |

**对文案的要求:** 屏 7 现在写的「每 24 小时自动更新一次」是在**承诺 iOS 给不了的保证**。改为:

> **「约每 24 小时更新一次,或在你打开 App 时」**

**另一个坑:** 那个流量数字是**机场服务端的账**,机场自己就有几分钟到几十分钟同步延迟,而且**本地正在跑的流量不会实时反映进去**。用户会疑惑"我刚下了 10GB 怎么数字没动"。所以**时间戳必须和数字绑定**,不能只在角落写一句。

### 1.6 ⛔ 内核明确做不到 — **设计上不要承诺**

| 想要的效果 | 现实 |
|---|---|
| iOS 上按 App 开关代理 | **做不到**(见 1.2 C 类) |
| 自动挑"能解锁 Netflix"的线路 | 内核**只测延迟,不测解锁**。要做得自建探测系统 |
| 自动挑"带宽最大"的线路 | **只有延迟,没有带宽概念** |
| 又要自动选最快、又要 IP 不变 | **互斥,二选一**(见 1.3) |
| 屏蔽视频里的贴片广告 | 只能**按域名**拦。YouTube 广告和正片同域名,拦了正片也没了 |

## 2. 现有设计稿的问题

### ✅ P0-1 「固定」语义矛盾 —— **屏 14 v2 已修复,以下保留为记录**

> **v2 复核结论:改对了。** 新稿把「指定地区」明确标注为"自动连延迟最低的线路 · 会自动切换(IP 会变)",把「锁定具体线路」单列并注明"只用这一条,出口 IP 不变",胶囊也变成了「🇺🇸 美国 · 洛杉矶-01 🔒」。语义已经自洽。
>
> **仅剩一个小问题(P2):** 当前选中的是「锁定具体线路 · 美国 洛杉矶-01」,但上方地区网格里的「美国」没有任何视觉呼应,用户会问"那我到底选没选美国?"。
> **建议:** 给「美国」加一个**弱高亮**(如浅色描边),表示"你锁定的线路属于这个地区",但**不要用与"选中地区自动"相同的选中态**(否则两处都亮,像选了两个)。

**以下为原始问题记录 ——**

**位置:** 屏 13 胶囊「🇺🇸 美国 · 固定」/ 屏 14

**问题:**
屏 13 的胶囊写「美国 · **固定**」,但屏 14 展开后当前值是「美国 · **地区内自动最快**」。

"地区内自动最快"意味着它会在美国的节点之间**定期测速、自动换节点** → **出口 IP 会变**。

于是屏 14 底部那句提示 **"AI 服务对 IP 变动敏感,固定地区可减少反复验证"** 兑现不了:IP 照样在变,Claude / ChatGPT 该弹验证还是弹。

而用户看到「固定」两个字,理解的是 **"我的 IP 不会变"**。这是硬性预期差,会直接吃差评。

**改法(你的稿子里其实已经有正确答案了,只是默认值和命名错位):**

1. **AI 服务(Claude / ChatGPT / Gemini)默认改为「指定具体线路」**——锁单节点才是真固定。胶囊显示成 `🇺🇸 美国 · 洛杉矶-01`
2. **"地区内自动最快"的胶囊改名为「美国 · 自动」**,**禁止使用"固定"二字**
3. 只有锁定单条线路时,才允许出现"固定"字样
4. 提示语相应改成:**"AI 服务对 IP 变动敏感,锁定单条线路可减少反复验证"**

### 🔴 P0-2 「解锁」内核不提供 —— 必须改或立项

**位置:** 屏 13 / 屏 9 的 Netflix「🇸🇬 新加坡 · 解锁」

**问题:**
内核**只会测延迟,完全不知道哪个节点能解锁 Netflix**。能不能解锁是节点 IP 的属性,内核无从得知。

要实现"解锁"标签,必须我们自己做一套:定期拿每个节点去请求 Netflix 的探测接口 → 解析结果 → 给节点打标 → 缓存。这是一个**独立功能模块**,而且判定逻辑会随 Netflix 风控策略变化而失效,需要长期维护。

**改法(二选一):**

- **方案 A(推荐,先上线):** 胶囊降级为「🇸🇬 新加坡 · 自动」,**不出现"解锁"字样**
- **方案 B:** 立项做解锁探测。那么 UI 需要额外设计:探测中状态、探测失败状态、"该地区暂无可解锁线路"的空状态、上次探测时间

> ⚠️ 最糟的做法是:显示"解锁"但底层没有探测。用户点了却打不开 Netflix → 投诉。

---

### 🔴 P0-3 「应用」措辞误导 iOS 用户 —— 必须改

**位置:** 屏 9「智能分流」描述:**"国外应用走线路,国内应用直连"**

**问题:**
如 1.2 C 类所述,iOS 上根本无法按 App 分流。用"**应用**"这个词,用户会理所当然地认为"Swell 能识别我的 App",然后来问:*"为什么我的 XX App 没走代理?"*

**改法:**
把所有面向用户的文案里的"应用"换成"**网站与服务**":

- ❌ 国外应用走线路,国内应用直连
- ✅ **国外网站与服务走线路,国内直连**

同理检查:屏 9「全局代理」的 "国内应用可能变慢" → "**国内网站可能变慢**"

### 🟡 P1-4 模式 与 服务分流 的关系没有表达

**位置:** 屏 9

**问题:**
用户切到「**全局代理**」后,下面 Claude 的「美国 · 固定」还生效吗?

**两种都能实现,但必须由产品定死,并且 UI 要说清楚**,否则用户困惑:

- **选项 A:** 全局模式覆盖一切 → 服务分流全部失效
- **选项 B:** 全局模式只接管"跟随智能分流"的服务,已显式指定的服务仍按指定走

**改法:**

- 若选 A:切到全局代理/全局直连时,「服务分流」整块**置灰**,并显示一行说明:**"当前模式下服务分流不生效"**
- 若选 B:在「服务分流」标题旁加说明:**"已指定的服务不受模式影响"**

---

### 🟡 P1-5 规则顺序被 UI 隐藏了,但底层是有序的

**位置:** 屏 9「+ 添加服务或网站」

**问题:**
内核是**从上到下第一条命中生效**(见 1.1)。但你的新 UI 是分组卡片,**完全没有顺序概念**——这对易用性是好事,但会产生冲突:

用户手动添加了 `youtube.com` → 直连,而内置的 YouTube 服务规则是 → 自动最快。**两条都能命中 YouTube,谁赢取决于生成顺序**,用户无法理解为什么。

**改法:**

1. 产品侧定死优先级(建议:**用户自定义 > 拦截 > 具体服务 > 地区/兜底**)
2. UI 侧:当用户添加的网站与已有服务重叠时,给出提示:**"youtube.com 已被内置服务「YouTube」覆盖,你的设置将优先生效"**,并提供"改为编辑 YouTube 服务"的快捷入口

---

### 🟡 P1-6 地区识别会失败,需要兜底

**位置:** 屏 14「指定地区」

**问题:**
内核**没有"地区"概念**。把节点归类到"美国/日本/香港"完全靠我们**解析节点名称**(🇺🇸 / US / 美国 / 洛杉矶 …)。机场的命名千奇百怪,**一定会有节点归不了类,或者归错**。

**改法:**

1. 「更多」里需要有 **"未识别地区"** 分组,列出归不了类的节点
2. 允许用户**手动给节点打地区标签**(节点详情页)
3. 需要设计"**该地区暂无可用线路**"的空状态

---

### 🟡 P1-7 「自动最快」= 延迟,不等于"流畅"

**位置:** 屏 9 / 屏 13 的 YouTube「⚡ 自动最快」

**问题:**
"最快"是按**延迟(ping)**选的,**不代表带宽大**。视频场景用户期待的是"不卡",但低延迟节点未必带宽好——可能出现"显示最快但看 4K 卡顿"。

**改法(轻量):**
胶囊文案改成「**⚡ 自动**」,在详情页说明"自动连接延迟最低的线路"。**避免用"最快"暗示带宽**。

## 3. 建议新增:内核白送的能力,现在没用上

这些 iOS 上都能实现(见 1.2 B 类),成本低、差异化强,建议 Design 考虑:

| 能力 | 可能的 UI |
|---|---|
| **按网络类型分流** | "使用蜂窝数据时自动直连"开关(省流量) |
| **低数据模式感知** | 系统开启低数据模式时自动降级 |
| **按 Wi-Fi 名称分流** | "在这个 Wi-Fi 下总是直连" —— **家里/公司 Wi-Fi 一键直连,体验很好** |
| 是否计费网络 | 个人热点下自动省流 |

> 「在家 Wi-Fi 自动直连」这个功能对国内用户很实用,而且竞品做的不多。

## 3.5 服务清单:默认预设 vs 推荐库

### 判断标准(重要)

> **预设的门槛不是"常用",而是"不预设就会出问题"。**

每多一条默认预设,主页清单就长一分,就离"配置清单"近一步。而 **常用 ≠ 需要特别设置** —— 智能分流已经能把绝大多数常用服务走通,预设它们等于告诉用户"这条我特意帮你设了",实际啥也没多干。

### 默认预设 —— 只有 5 条,都有硬理由

这些**开箱即生效**,出现在主页的「服务例外」列表里:

| 服务 | 默认去向 | 为什么必须预设 |
|---|---|---|
| Claude | 🔒 锁定单条线路 | IP 变动触发反复验证 |
| ChatGPT | 🔒 锁定单条线路 | 同上 |
| Gemini | 🔒 锁定单条线路 | 同上 |
| **X (Twitter)** | 🔒 锁定单条线路 | **风控最激进之一,IP 跳动触发手机/邮箱验证,严重时锁号** |
| **TikTok** | 🔒 锁定(默认日本,可改) | **地区直接决定内容池;IP 乱跳会让推荐混乱、账号异常** |

### 明确**不**预设的(以及原因)

| 服务 | 为什么不预设 |
|---|---|
| **YouTube** | 智能分流已经走通,地区不影响能不能看。看视频不需要会话 IP 一致性,换线路无感。预设 = 占位置不干活 |
| **Telegram** | 智能分流已覆盖,而且它**不挑地区**,任何能通的线路都一样 |
| **Netflix** | 我们没有解锁检测能力(见 P0-2),预设 = 空承诺 |

## 3.6 落地检测:完整逻辑

### 为什么需要它

「地区」是靠**解析线路名**猜的(见 1.4)。机场的名字**会骗人**——名字写「美国-高速03」,实际落地在香港。用户把 Claude 锁到这条线上,以为锁了美国 IP,其实没有。

**落地检测** = 通过该线路访问一次我们自建的 IP 回显接口,拿回真实落地国家。

### 产品决策

| 决策 | 结论 | 理由 |
|---|---|---|
| 默认状态 | **关闭** | 要向外发请求(隐私);48 条线路 = 48 次请求 |
| 检测时机 | 默认**仅手动** | 同上 |
| IP 回显服务 | **自建** | 不把用户流量导给第三方,可写进隐私政策当卖点 |
| 主入口 | **不是设置页** | 见下 |

### 关键:入口不能只埋在设置里

默认关的功能,**90% 用户永远不会去设置里翻**。真正的入口应该是 **场景内引导**:

用户在「**为 Claude 锁定线路**」——这个**唯一真正需要准确落地**的场景——展示一张卡:

> 线路地区目前按名称推测,可能与真实落地不符。AI 服务看到的是落地 IP。
> **[ 检测这 12 条美国线路 ]**

**只检测当前地区的 12 条,不是全部 48 条。在用户需要的那一刻,只付他需要的那部分代价。** 设置里的开关只是给已经理解这功能的人用的全局总闸。

### 状态如何显示 —— **长在旗帜上,不占第二行**

线路行里**已经有旗帜了**,而旗帜就是地区信息。落地检测的结果**不是新增信息,是对旗帜的确认或纠正**。所以不要再加"地区胶囊",状态直接长在旗帜上:

| 状态 | 表现 |
|---|---|
| 检测关闭(默认) | 旗帜正常,**无任何标记**。列表与现在完全一致 |
| 实测一致 | 旗帜右上角小 ✓ 角标 |
| **实测不符** | **旗帜直接换成实测国家** + ⚠ 角标 + 第二行「名称写美国,实测落地香港」+ 警告底色 |
| 未识别 | 旗位显示灰色占位 |
| 检测中 | 旗位显示 spinner |

> **关闭时的降级必须彻底:** 不出现任何检测相关 UI,不要有"未检测"灰标——那是在惩罚没开这功能的用户。

### 在分流模块里的后果

用户为 Claude 选美国线路时,那条**名字叫美国、实测香港**的必须**从列表中排除**,并明确告知:

> ⚠️「美国-高速03」名称含美国,但实测落地香港,已从本列表排除

不说的话用户会疑惑"我明明有条美国-高速03,怎么不在列表里"。**这是整个功能落到用户价值上的那一下。**

## 3.7 节点列表:双行 + 订阅头

### 为什么改双行

单行放不下(旗帜 + 名称 + 协议 + 倍率 + 延迟 + 落地状态)。但**双行有代价**:48 条线路,单行约看 14 条,双行只能看 10 条,**密度掉 30%**。

**所以第二行必须真的有东西,不能只放个 `Trojan` 凑数。** 第二行承担三件事,**平时灰、异常才变色**:

| 内容 | 平时 | 异常 |
|---|---|---|
| 协议 | 灰色 `Trojan` | — |
| **倍率** | 不显示(1x 或未写) | **`2x 倍率` 标黄** |
| 落地 | 不显示 | `名称写美国,实测落地香港` 标黄 |

> **倍率要提级。** 它直接决定扣多少流量,比协议重要得多。现在做成小标签容易被忽略,应该在第二行用警告色。

### 订阅分组头 —— 流量与到期提到列表里

不用点进「管理」才能看到。分组头四种状态:

| 状态 | 表现 |
|---|---|
| **正常** | `蓝鲸云 36 条` + 右侧 `剩 61.5 GB · 47 天` + **一条 2px 细进度条**(兼做分组分隔线,不额外占高度) |
| **告警** | 剩余 **<10% 或 <7 天** → 数字与进度条转红 + ⚠ 图标。**这是用户唯一真正在意的时刻** |
| **自建**(无元数据) | 只显示 `我的自建 3 条`。**进度条与到期整块隐藏** |
| **数据陈旧**(拉取失败) | 数字**灰化**,下方 `3 天前的数据 · 点击重试`。**不假装是新的** |

【B. 节点列表的状态显示 —— 状态长在旗帜上,不占第二行】
线路行里已经有旗帜,旗帜就是地区信息。检测结果不是新增信息,

### 提示词 12 — 订阅管理页的自建降级态与文案(屏 7)

```
修改 Swell「订阅管理」页(屏 7)。

问题一:文案承诺了 iOS 给不了的保证
现在写「每 24 小时自动更新一次」。但 iOS 没有可靠的后台定时任务,
App 被挂起时不会自己去拉,系统的后台刷新不保证准时且用户可关闭。
实际更新几乎都发生在用户打开 App 时。
→ 改为:「约每 24 小时更新一次,或在你打开 App 时」

问题二:缺少自建订阅的降级态
自建线路(如 CF Workers)拿不到流量配额和到期日——那是机场的商业模型,
不是技术属性。现在的卡片设计只有「有流量有到期」这一种完整态。
→ 请补一个精简态卡片:只有订阅名 + N 条线路 + 上次更新时间,
  流量进度条和到期整块隐藏。不要显示「未知」「—」或空进度条。

问题三:时间戳要和数字绑定
流量数字是机场服务端的账,机场自己有几分钟到几十分钟延迟,
且本地正在跑的流量不会实时反映进去。用户会疑惑
「我刚下了 10GB 怎么数字没动」。
→ 「2 小时前更新」不能只放在角落,要和流量数字在视觉上关联

另外请补一个「数据陈旧」态:拉取失败超过 24 小时时,
数字灰化 + 「3 天前的数据 · 点击重试」。
```
